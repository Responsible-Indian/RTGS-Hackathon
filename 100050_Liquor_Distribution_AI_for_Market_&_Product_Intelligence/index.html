Core Problem Areas

  1. 01

    Forecasting in the Dark

    Demand prediction relies on historical patterns and intuition rather than scientific models. Without AI-driven forecasting, APSBCL cannot anticipate seasonal surges, regional variations, or consumption trend shiftsleading to reactive rather than proactive inventory decisions.

    Inventory miscalculations across districts
    Metric: Forecast Accuracy
  2. 02

    Consumers Without Profiles

    Despite transacting with thousands daily, there's no systematic understanding of who buys what, when, or why. Purchase patterns, preference clusters, and behavioral segments remain invisiblemaking targeted inventory and marketing approaches difficult.

    Misaligned product-market fit
    Metric: Basket Analysis
  3. 03

    The Inventory Seesaw

    Some outlets accumulate excess stock while others run dry. Without intelligent redistribution signals, high-demand products sit in low-traffic locations while busy outlets face stockouts. Capital gets locked in idle inventory as sales opportunities are missed elsewhere.

    Lost sales and capital inefficiency
    Metric: Stockout Rate
  4. 04

    GPS Data Sitting Idle

    Delivery vehicles generate continuous location data, yet it rarely informs operational decisions. Route efficiency remains unoptimized, delivery times unpredictable, and supply chain bottlenecks difficult to identify. The data existsthe intelligence layer doesn't.

    Logistics costs and delivery delays
    Metric: On-Time Delivery
  5. 05

    SKU Sprawl Without Strategy

    Hundreds of SKUs compete for limited shelf space without performance-based rationalization. Slow-moving products occupy premium positions while high-demand items run out. There's no data-driven framework to optimize the product portfolio or phase out underperformers.

    Shelf inefficiency and working capital drag
    Metric: SKU Contribution
  6. 06

    Outlets Without Identity

    All 3,736+ outlets receive largely uniform treatment despite differences in location, footfall, consumer demographics, and sales potential. High-performers may be under-resourced while struggling outlets lack targeted interventionlimited segmentation means limited precision.

    Suboptimal resource allocation
    Metric: Outlet Performance
Root Cause Analysis

Where Intelligence
Breaks Down

The problems identified aren't failures of effortthey're symptoms of systemic friction. Here's why APSBCL's data never becomes operational intelligence.

Data flows in, but decisions flow out slowlythe intelligence layer that should connect them doesn't exist.

Why the Gaps Persist

01

Data Living in Silos

Cause: Fragmented sources Effect: No cross-modal features

Sales transactions, GPS logs, Suraksha App data, and outlet profiles exist in separate systems with no integration layer. Each data stream provides partial visibility, but without fusion, no system-wide intelligence can emerge.

No API contracts or shared IDs between Sales, GPS, Suraksha, Outlet master joins fail and cross-modal features can't be computed.

Multimodal insights remain invisible
02

Manual Intervention Trap

Cause: No automation Effect: Human bandwidth limits throughput

SKU allocation at depots still requires human judgment for each decision. Without automated recommendations, every distribution choice becomes a throughput bottleneckstaff capacity limits system capacity.

No daily forecast-to-allocation auto-feed depot staff must re-derive decisions from raw tables each cycle.

Decisions bottlenecked by human bandwidth
03

Missing Intelligence Layer

Cause: No ML infrastructure Effect: Data value unrealized

Raw data accumulatesGPS pings, transaction records, inventory countsbut there's no machine learning layer to transform it into forecasts, segments, or anomaly alerts.

No streaming/feature store models can't infer in near real-time; batch lags behind operational needs.

Data collected but value unrealized
04

No Outlet Taxonomy

Cause: Incomplete profiles Effect: Segmentation impossible

Outlet profiles lack standardized, complete featuresfootfall estimates, location types, consumer demographics are missing or inconsistent. Without a complete taxonomy, clustering algorithms can't segment outlets meaningfully.

No systematic outlet profiling process ground-truth verification absent; segmentation has no features to cluster on.

One-size-fits-all distribution persists
05

GPS Signal Without Semantics

Cause: No route baselines Effect: Deviations uninterpretable

GPS logs capture vehicle movement, but without baseline routes, POI models, and sampling normalization, deviations can't be distinguished from legitimate detours. Alerts lack context to determine pilferage risk vs. traffic reroutes.

No route baseline repository or POI context deviation alerts fire without validation logic; false positives overwhelm signal.

Route intelligence unactionable
06

Reactive Decision Cycles

Cause: No prediction branch Effect: Always behind

Current operations detect problems after they occurstockouts, route delays, demand spikes. There's no prediction branch in the decision cycle. By the time issues surface, the damage is done.

Forecasts not scheduled/event-driven; no exception calendar (festivals/dry days) only after-the-fact action.

Problems addressed after impact
07

Broken Feedback Loops

Cause: No outcome tracking Effect: No model improvement

When allocation decisions lead to stockouts or wastage, that outcome data doesn't flow back to improve future decisions. Alerts lack prioritization by severitysignal-to-noise prevents timely action, and corrections never retrain models.

No discrepancy reporting channel; no human-in-the-loop thresholds outcomes never retrain models or improve forecasts.

Past outcomes don't improve accuracy
08

Scale Without Infrastructure

Cause: No horizontal scaling Effect: Peak periods overwhelm

The operation spans thousands of outlets generating continuous data streams, yet there's no computational infrastructure to process it at scale. Festival spikes, election periodsvolume exceeds current processing capacity.

No horizontal scaling for peak periods; GPS stream not partitioned/aggregated dashboards lag behind reality.

Data volume overwhelms capacity

The Core Insight

These bottlenecks aren't independent failuresthey form a self-reinforcing system of inertia. Data fragmentation necessitates manual intervention. Manual processes can't scale, forcing reactive responses. Reactive operations prevent learning loops. Without learning, intelligence never accumulates to break the cycle. This is why APSBCL's problems persist despite available data and dedicated operational effort.

Section 06 User Flows

Stakeholder Journeys

End-to-end user experiences across five key personas in the AI-Enabled Market Intelligence Platform

Department Officer

Senior officials from Prohibition and Excise Department responsible for policy decisions, revenue targets, and strategic oversight of distribution operations across the state.

Review Statewide Performance & Approve Recommendations

Primary Flow
Technical Architecture
1
Access executive dashboard for statewide KPIs
Officer logs into the Superset-based executive dashboard via SSO (Keycloak). The dashboard displays real-time state-level metrics including total revenue, sales velocity by district, depot utilization rates, and SKU-wise performance. Data is fetched through Kong API Gateway with role-based filtering ensuring officers see only their jurisdiction's data. Redis caching ensures sub-second dashboard load times.
Tech: Apache Superset Kong Gateway Keycloak SSO Redis Cache
2
Review AI-generated SKU recommendations
The system presents ML-driven recommendations for SKU portfolio optimization. XGBoost models analyze historical sales patterns, seasonal trends, and consumer preferences from Suraksha App data to suggest which SKUs to introduce, discontinue, or adjust pricing. Each recommendation includes projected revenue impact (% change), confidence intervals, and supporting visualizations showing demand forecasts.
Tech: XGBoost Prophet Forecasting Delta Lake SHAP Explainability
3
Approve policy changes with audit trail
Officer reviews and approves recommended policy changes through a structured approval workflow. Upon approval, the system logs the decision with full audit trail including officer ID, timestamp, decision rationale, and affected entities. PostgreSQL stores immutable audit records while Keycloak enforces that only authorized officers (Commissioner/Joint Commissioner level) can approve state-wide policy changes.
Tech: PostgreSQL Audit Keycloak RBAC Immutable Logging Digital Signatures
4
Monitor GPS compliance and route deviations
Real-time GPS telemetry from delivery vehicles streams through Apache Flink for continuous anomaly detection. DBSCAN clustering identifies unusual route patterns, unauthorized stops, or geofence violations. The dashboard displays a live map with vehicle positions, flagging any deviations with severity scores. Officers can drill down into specific alerts to view complete GPS trails and delivery timestamps.
Tech: Apache Flink DBSCAN Clustering Kafka Streams PostGIS Geofencing

Investigate & Act on High-Priority Alerts

Alert Flow
Technical Architecture
1
Receive escalated high-priority alert
Alert Engine detects critical anomalies (route deviation >2km, unscheduled stop >30min, GPS signal loss) and escalates to senior officers. Multi-channel notification via SMS, email, and push notification includes alert severity, vehicle ID, driver details, and last known location. Redis Pub/Sub ensures real-time delivery with <500ms latency. Officers receive deep-links to investigation dashboard.
Tech: Alert Engine Redis Pub/Sub SMS Gateway Firebase Push
2
Drill into alert details with GPS evidence
Investigation dashboard renders complete GPS trail on interactive map showing planned vs actual route. GraphQL queries fetch historical trips for same driver/vehicle for pattern analysis. Evidence panel displays photos captured by driver (if deviation was logged), delivery confirmation timestamps, and vehicle sensor data. Geofence violation zones are highlighted with timestamp markers.
Tech: Apache Superset GraphQL API Mapbox GL PostGIS
3
Authorize enforcement action with audit trail
Officer selects enforcement action from predefined options: issue warning, suspend driver, initiate inquiry, or mark as false positive. Decision requires digital signature via Keycloak. Complete audit trail captures decision rationale, supporting evidence reviewed, and officer's jurisdiction confirmation. Downstream systems (HR, vehicle dispatch) are automatically notified of suspension actions.
Tech: Keycloak RBAC PostgreSQL Audit Event-Driven Actions Digital Signatures

APSBCL Depot Staff

Operational personnel at wholesale depots responsible for daily SKU allocation, stock management, vehicle dispatch, and coordinating deliveries to retail outlets.

Execute Daily SKU Allocation Based on AI Recommendations

Primary Flow
Technical Architecture
1
View daily allocation dashboard
Depot staff logs into the allocation dashboard showing prioritized list of outlets requiring replenishment. Each outlet displays current stock levels, days-of-inventory remaining, and AI-recommended quantities per SKU. The dashboard is pre-cached in Redis at 5 AM daily after overnight batch processing, ensuring instant load times. Color-coded urgency indicators highlight outlets at risk of stock-out within 24-48 hours.
Tech: Apache Superset Redis Cache Morning Batch Job Priority Scoring
2
Review 7-day demand forecasts per SKU-outlet
Staff clicks on any outlet to view detailed 7-day demand forecasts for each SKU. Prophet models generate daily predictions with 80% and 95% confidence intervals displayed as shaded bands on time-series charts. XGBoost provides feature importance showing which factors (festivals, weather, day-of-week, historical patterns) are driving the prediction. Staff can toggle between pessimistic/optimistic scenarios.
Tech: Prophet XGBoost Confidence Intervals Scenario Analysis
3
Confirm or adjust allocation quantities
Staff can accept AI recommendations or manually adjust quantities based on local knowledge (e.g., upcoming local event not in system). When adjusting, staff selects a reason code (local event, vehicle capacity, supplier delay) which feeds back into the ML training pipeline for continuous improvement. All adjustments are logged with staff ID, timestamp, original vs modified values for model accuracy tracking.
Tech: Kong API PostgreSQL Feedback Loop Reason Codes
4
Generate optimized dispatch orders
Upon confirmation, OR-Tools optimization engine generates dispatch orders with: (1) optimized vehicle loading plan respecting weight/volume constraints, (2) route sequence minimizing total travel distance via TSP solver, (3) driver-vehicle assignment based on availability and route familiarity. Dispatch orders include item-wise loading sequence, expected delivery times, and barcode manifests for each vehicle.
Tech: Google OR-Tools VRP Solver Bin Packing TSP Optimization
5
Monitor real-time delivery progress
Live tracking dashboard shows all dispatched vehicles on a map with real-time GPS positions updated every 30 seconds via Flink stream processing. Each vehicle card displays current location, completed/pending deliveries, ETA to next stop, and route compliance percentage. Redis stores latest positions for instant retrieval. Staff receives proactive alerts if any vehicle is running >30 minutes behind schedule.
Tech: Apache Flink GPS Streaming Redis ETA Prediction

Respond to Stock Imbalance Alert

Alert Flow
Technical Architecture
1
Receive stock imbalance notification
Alert Engine continuously monitors inventory levels against demand forecasts. When an outlet's projected stock falls below safety threshold (typically 48-hour coverage), an urgent notification is triggered. SMS and app push notification includes outlet name, affected SKUs, current stock, predicted stock-out date, and urgency level (High/Critical). Alert contains one-tap link to rebalancing dashboard.
Tech: Alert Engine SMS Gateway Safety Stock Algorithm Firebase Push
2
View recommended rebalancing actions
Dashboard presents AI-generated rebalancing options: (1) Inter-outlet transfer from nearby outlets with excess inventory, (2) Emergency dispatch from depot, or (3) Supplier expedite request. Each option shows estimated cost, time-to-resolution, and revenue-at-risk calculation. System identifies donor outlets with >7 days inventory for transfer candidates based on geographic proximity and SKU availability.
Tech: Apache Superset Delta Lake Cost-Benefit Analysis Geographic Routing
3
Approve and execute rebalancing
Staff selects preferred rebalancing option and approves execution. System immediately generates emergency dispatch order with optimized route to minimize transfer time. For inter-outlet transfers, both source and destination outlets receive SMS notifications with expected pickup/delivery times. OR-Tools calculates fastest route considering current traffic conditions. Digital dispatch note is generated for driver.
Tech: Google OR-Tools Route Optimizer Real-time Traffic Dispatch Automation
4
Confirm delivery and resolve alert
Upon delivery confirmation via GPS geofence trigger and digital signature, inventory levels are automatically updated in real-time. Flink stream processor updates Delta Lake within seconds. Original alert status transitions to "Resolved" with complete audit trail showing time-to-resolution, quantities transferred, and staff actions. KPI dashboard updates to reflect improved service level.
Tech: Apache Flink GPS Geofencing PostgreSQL Auto-Resolution

Analytics Team

Data analysts and business intelligence professionals who perform deep analysis, create custom reports, and develop insights for strategic planning.

Perform Deep-Dive Analysis on Consumer Behavior Trends

Analytical Flow
Technical Architecture
1
Access analytical workspace with SQL interface
Analyst logs into Superset SQL Lab with role-based access to anonymized data warehouse. Presto query engine provides federated access across Delta Lake tables (sales, inventory, consumer segments) with sub-second query times. All PII is pre-masked; analyst sees only hashed consumer IDs and aggregated demographics. Query history and saved queries enable reproducible analysis workflows.
Tech: Apache Superset SQL Lab Presto PII Masking Delta Lake
2
Execute custom queries on consumer segments
Analyst queries pre-computed consumer segmentation tables derived from Suraksha App data. K-Means clustering has identified distinct consumer personas (premium buyers, value seekers, occasional purchasers). RFM (Recency, Frequency, Monetary) analysis provides customer lifetime value scores. Analyst can slice data by segment, geography, time period, and SKU category to identify emerging trends and patterns.
Tech: K-Means Clustering RFM Analysis Customer Segmentation Cohort Analysis
3
Cross-reference with outlet segmentation
Analyst joins consumer segment data with outlet classification (urban premium, rural value, highway transit) to understand which outlet types serve which consumer segments. Identifies underserved combinations (e.g., premium SKUs at outlets with high-value consumer traffic). Market basket analysis reveals cross-selling opportunities. Heatmaps visualize geographic distribution of segment concentrations.
Tech: Delta Lake Joins Market Basket Analysis Geographic Analytics Heatmap Viz
4
Generate custom reports with recommendations
Analyst builds interactive dashboard with key findings using Superset's drag-and-drop chart builder. Report includes executive summary, supporting visualizations (bar charts, trend lines, geo maps), and data-driven recommendations. Export options include PDF for executives, CSV for downstream analysis, and scheduled email delivery. Dashboard can be shared with stakeholders via secure links with appropriate access controls.
Tech: Apache Superset Export API (PDF/CSV) Scheduled Reports Access Control

Validate and Tune Demand Forecasting Models

ML Ops Flow
Technical Architecture
1
Access model performance dashboard
MLflow UI provides comprehensive model monitoring with key metrics: MAPE (Mean Absolute Percentage Error) targeting <15%, MAE (Mean Absolute Error), and forecast accuracy at 7-day horizon. Dashboard segments performance by SKU category, outlet type, and district. Trend charts show model drift over time. Automated alerts trigger when any segment falls below accuracy threshold (85%).
Tech: MLflow Metrics API MAPE/MAE Tracking Drift Detection
2
Identify underperforming model segments
Analyst drills into model registry to find specific SKU-outlet combinations with accuracy below threshold. MLflow Tracking shows experiment history with parameters used for each training run. Delta Lake provides actual vs predicted comparison data for forensic analysis. Analyst can filter by error magnitude, geographic region, or SKU attributes to isolate problem areas requiring attention.
Tech: MLflow Tracking Delta Lake Experiment Comparison Error Analysis
3
Analyze feature importance and residual patterns
SHAP (SHapley Additive exPlanations) values reveal which features drive predictions for problematic segments. Prophet decomposition shows trend, seasonality, and holiday effects separately. Residual plots identify systematic bias patterns. Analyst examines if errors correlate with external factors (weather, local events) not currently in feature set. Identifies opportunities for feature engineering improvements.
Tech: SHAP Explainability Prophet Decomposition Residual Analysis Feature Engineering
4
Trigger model retraining with A/B staging
Analyst initiates retraining job with adjusted hyperparameters or additional features via MLflow. New model version is automatically registered in Model Registry with lineage tracking. Shadow deployment serves predictions in parallel with production model for validation. After 7-day A/B test with statistical significance, analyst promotes challenger model to production if metrics improve. Full rollback capability ensures safety.
Tech: MLflow Model Registry A/B Testing Shadow Deployment Canary Release

Vehicle Driver

Drivers of delivery vehicles transporting liquor from depots to retail outlets, equipped with GPS-enabled devices.

Complete Assigned Delivery Route

Primary Flow
Technical Architecture
1
Receive route assignment via mobile app
Driver opens the delivery app (Flutter-based) which displays today's assigned route with optimized sequence calculated by OR-Tools VRP solver. Each stop shows outlet name, address, delivery quantities by SKU, expected arrival time, and special instructions. Turn-by-turn navigation integrates with Google Maps. Route considers traffic predictions, vehicle capacity, and delivery time windows for each outlet.
Tech: Flutter Mobile App Google OR-Tools Google Maps SDK Traffic Prediction
2
Start trip and enable GPS tracking
Driver taps "Start Trip" which activates continuous GPS tracking via device sensors. Location pings are transmitted every 30 seconds via Kafka message queue, ensuring no data loss even during connectivity gaps (offline queue with automatic sync). Backend receives coordinates with timestamp, speed, and heading. Flink processes stream in real-time for immediate anomaly detection and ETA recalculation.
Tech: GPS Sensor Apache Kafka Offline Queue Apache Flink
3
Complete deliveries at each outlet
Upon arrival at outlet (detected via geofence), driver scans dispatch barcode and outlet QR code for verification. App displays items to deliver with checkbox confirmation. Outlet manager signs digitally on device screen. Photos of delivered goods can be captured as evidence. Delivery record includes exact GPS coordinates, timestamp, and recipient signatureall synchronized to PostgreSQL immediately.
Tech: Geofencing Barcode/QR Scanner Digital Signature PostgreSQL
4
Return to depot with compliance report
Driver returns to depot and taps "End Trip" upon entering depot geofence. System automatically generates trip summary: total distance, time taken, fuel efficiency estimate, and route compliance percentage (planned vs actual route adherence). Any deviations are flagged with reasons if provided. Driver performance score updates in real-time. Trip data feeds into analytics for continuous route optimization.
Tech: Flink GPS Analytics Route Compliance Score Performance Metrics Delta Lake

Handle Legitimate Route Deviation

Exception Flow
Technical Architecture
1
Encounter road closure or emergency
Driver encounters unexpected road blockage (accident, construction, flooding, VIP movement). The app detects route deviation in real-time as GPS coordinates diverge from planned path. Mobile app immediately displays prompt: "Route Deviation Detected - Please provide reason." Flink stream processor flags the deviation within seconds, but holds escalation pending driver input.
Tech: GPS Detection Apache Flink Real-time Monitoring Deviation Alert
2
Log deviation reason with photo evidence
Driver selects reason code from predefined list: Road Closure, Accident, Weather, Vehicle Issue, or Other. App requires geotagged photo evidence showing the obstruction (timestamp and location embedded in EXIF data). Driver can add voice note in Telugu/English for additional context. All evidence uploads to S3 with automatic retry for poor connectivity. Deviation record links to trip ID for audit trail.
Tech: Mobile App S3 Storage Geotagged Photos Voice Recording
3
System evaluates deviation legitimacy
ML classifier (trained on historical deviation patterns) evaluates legitimacy score based on: (1) Cross-reference with traffic APIs for reported incidents, (2) DBSCAN clustering to check if other vehicles deviated similarly, (3) Time-of-day and seasonal patterns, (4) Driver's historical deviation frequency. High legitimacy score (>0.8) auto-approves with no escalation. Low scores escalate to supervisor for review. Photo evidence is stored for audit.
Tech: DBSCAN Clustering ML Classifier Traffic API Cross-ref Auto-Approval Engine

System Administrator

IT personnel responsible for system configuration, user management, and technical operations.

Configure New Depot and Outlets in System

Setup Flow
Technical Architecture
1
Add depot configuration via admin portal
Administrator logs into the admin portal with elevated privileges. Creates new depot record with: depot code, name, address, GPS coordinates (lat/long), serving district boundaries (GeoJSON polygon), storage capacity (cases), vehicle fleet count, and operating hours. Validation rules ensure unique depot codes and valid geographic boundaries. PostgreSQL stores configuration with audit trail of who created and when.
Tech: Admin Portal (React) PostgreSQL PostGIS Audit Logging
2
Import outlet master data via CSV
Administrator uploads CSV file containing outlet details: outlet code, license number, name, address, GPS coordinates, category (urban/rural/highway), storage capacity, and assigned depot. System validates each row: checks for duplicate codes, validates GPS within state boundaries, verifies license format, and ensures depot assignment exists. Import summary shows successful records and rejected rows with error reasons.
Tech: CSV Parser Data Validation Engine Batch Insert Error Reporting
3
Configure data integration pipelines
System automatically provisions Kafka topics for new depot (sales events, inventory updates, GPS telemetry). Flink jobs are configured to include new depot in stream processing. Delta Lake tables are updated to partition by new depot for query optimization. Demand forecasting models are scheduled to train on historical data from similar outlets to bootstrap predictions. All pipeline configurations are versioned in GitOps repository.
Tech: Apache Kafka Apache Flink Delta Lake GitOps (ArgoCD)
4
Assign user access permissions
Administrator creates user accounts in Keycloak for depot staff with appropriate RBAC roles: Depot Manager (full access), Dispatch Staff (route assignment), Inventory Clerk (stock updates). Role definitions include depot-level scope limiting data visibility to assigned depot only. mTLS certificates are generated for service-to-service authentication. Users receive credentials via secure email with mandatory password reset on first login.
Tech: Keycloak IAM RBAC Roles mTLS Certificates Secure Onboarding
